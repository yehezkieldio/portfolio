This is the two-laptop setup I use when my Windows 11 laptop has the upstream Wi-Fi connection and my Linux laptop is plugged into it over Ethernet. Windows acts as the small NAT gateway, and the Linux laptop gets a stable wired dev link without needing to join the upstream network directly.

The practical benefit is boring in a good way: the Linux laptop gets internet from Windows, local development is easier over Ethernet, and I can still expose a Next.js dev server to a phone on the same Wi-Fi by forwarding one port through Windows.

The shape:

- Windows 11 stays connected to the real Wi-Fi network.
- Windows Ethernet becomes the private gateway at `192.168.137.1`.
- The Linux laptop sits behind Windows at `192.168.137.2`.
- WinNAT gives the Linux laptop outbound internet through Windows.
- A static WinNAT mapping forwards Windows port `3000` to the Linux laptop's port `3000`.

## Topology

| Device | Interface | Role | Address |
| --- | --- | --- | --- |
| Windows 11 | Wi-Fi | Upstream internet | DHCP from router |
| Windows 11 | Ethernet | NAT gateway | `192.168.137.1/24` |
| WinNAT | `ArchNAT` | NAT network | `192.168.137.0/24` |
| Linux laptop | Ethernet | NAT client | `192.168.137.2/24` |

Normal outbound traffic flows from Linux to Windows, then from Windows through Wi-Fi to the internet. Phone access goes the other direction: phone to the Windows Wi-Fi IP on port `3000`, then Windows forwards that request over Ethernet to the Linux laptop.

## Windows Setup

On Windows, I set the Ethernet adapter manually:

- IP address: `192.168.137.1`
- Subnet mask: `255.255.255.0`
- Default gateway: blank
- DNS: blank

Then I create the WinNAT network from an elevated Windows PowerShell 5.1 session:

```powershell
Get-NetIPAddress -InterfaceAlias "Ethernet"

New-NetNat `
  -Name "ArchNAT" `
  -InternalIPInterfaceAddressPrefix 192.168.137.0/24

Get-NetNat
```

For the dev server, I add a static mapping:

```powershell
Add-NetNatStaticMapping `
  -NatName "ArchNAT" `
  -Protocol TCP `
  -ExternalIPAddress 0.0.0.0 `
  -ExternalPort 3000 `
  -InternalIPAddress 192.168.137.2 `
  -InternalPort 3000

Get-NetNatStaticMapping
```

Then I allow the inbound port through Windows Firewall:

```powershell
New-NetFirewallRule `
  -DisplayName "Allow TCP 3000 to Linux dev laptop" `
  -Direction Inbound `
  -Protocol TCP `
  -LocalPort 3000 `
  -Action Allow
```

For long sessions, I keep Windows awake while plugged in and disable power saving on the Wi-Fi and Ethernet adapters. If Windows sleeps or powers down the adapter, the Linux side looks broken even though the Linux config is fine.

## Linux Setup

On Linux, I use a static NetworkManager profile for the Ethernet link:

```bash
nmcli con add type ethernet ifname enp3s0 \
  con-name win-nat \
  ipv4.addresses 192.168.137.2/24 \
  ipv4.gateway 192.168.137.1 \
  ipv4.dns "1.1.1.1 1.0.0.1" \
  ipv4.method manual

nmcli con up win-nat
```

The interface name may differ. I check it with:

```bash
ip link
```

For Next.js, the dev server needs to bind to all interfaces:

```bash
npm run dev -- -H 0.0.0.0 -p 3000
```

The package manager does not matter. The important part is `0.0.0.0`; binding only to localhost means Windows cannot reach the server over Ethernet.

## Phone Access

I access the site from the phone through the Windows Wi-Fi address, not the Linux Ethernet address. First I find the Windows Wi-Fi IP:

```powershell
ipconfig
```

Then I open this from a phone on the same Wi-Fi:

```txt
http://<windows-wifi-ip>:3000
```

The phone should not use `192.168.137.2`. That address only exists on the private Ethernet side behind Windows. The phone reaches Windows first, and Windows forwards the request.

## Quick Checks

From Linux:

```bash
ping -c 3 192.168.137.1
curl -I http://192.168.137.2:3000
```

From Windows:

```powershell
Test-NetConnection 192.168.137.2 -Port 3000
Get-NetNatStaticMapping
```

From the phone:

```txt
http://<windows-wifi-ip>:3000
```

If Windows can reach Linux but the phone cannot reach Windows, I check the firewall rule and confirm the phone is on the same Wi-Fi network as Windows.

## Reset

Remove the port forward:

```powershell
Remove-NetNatStaticMapping -NatName "ArchNAT" -ExternalIPAddress 0.0.0.0 -ExternalPort 3000 -Protocol TCP
```

Remove the NAT:

```powershell
Remove-NetNat -Name "ArchNAT"
```

Remove the Linux NetworkManager profile:

```bash
nmcli con delete win-nat
```

## The Important Detail

WinNAT is outbound-only by default. That is enough for the Linux laptop to use the internet through Windows, but it is not enough for a phone to open the Linux dev server. The static mapping is the step that makes port `3000` reachable from the Wi-Fi side.
