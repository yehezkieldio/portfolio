## Why It Exists

Cubic is an internal operating system for a billiard venue. It connects the people taking orders, the kitchen fulfilling them, the admin desk monitoring and printing receipts, and the owner reading the business through sales, stock, and operational history.

The product is not trying to be a generic POS template. It is shaped around the real rhythms of the venue: Marka staff enter orders by table or customer name, kitchen staff resolve items one by one, admin watches live orders and receipts, and the owner audits money, inventory, staff performance, menu activity, and exceptions after the fact.

That made Cubic less of a dashboard project and more of a correctness project. The hard part is not rendering a table of orders. The hard part is making sure every status change, receipt, stock movement, financial snapshot, date filter, and production rollout agrees with the business rules after weeks of messy real usage.

## Product Surfaces

Cubic is split into role-specific surfaces rather than one universal admin panel:

- Marka creates orders from a dense menu/cart flow.
- Kitchen manages live fulfillment, menu availability, packets, stock entry, and item exceptions.
- Admin monitors active orders, manages Marka accounts, controls receipt printing, and handles operational settings.
- Owner reads analytics, histories, stock, terminals, settings, and system diagnostics.

That role split matters because each surface has a different job. Marka needs a fast, low-friction ordering path. Kitchen needs stable live boards and clear item state. Admin needs monitoring and printing to be reliable under pressure. Owner needs bounded reporting and audit trails that can survive scrutiny.

The UI follows that same operational posture: compact tables, dense controls, persistent navigation, tablet-aware owner tables, route-local filters, and live views that prioritize scanning over decoration.

## Order Lifecycle

The core flow starts when a Marka submits an order through `POST /api/orders`. Cubic snapshots menu identity, prices, PPN rate, per-cycle order number, table or customer name, requested hot/cold drink variants, and the initial financial totals.

From there, kitchen and admin screens update through SSE invalidation plus polling. A new order event is emitted as soon as the committed order row and items exist, while analytics and search read-model maintenance is scheduled afterward. That keeps the operational board responsive without pretending derived reporting data is the same thing as the source transaction.

Kitchen item actions are explicit:

- `done` means the item was fulfilled and stock sales movement is written.
- `kosong` means staff-facing Batal, a lost-sale exception with no stock movement.
- `cancelled` means the item is terminally cancelled, also without stock movement.
- `Tutup Order` closes the parent order and handles any remaining pending items through an explicit override path.

That separation is important. Completing an item is not the same thing as closing an order. Marking an item as unavailable is not the same thing as selling it. Cubic keeps those differences visible in the data model instead of smoothing them into one convenient status.

## Inventory

Cubic treats stock as a ledger first and a read model second. The source of truth is `stok_logs`; `stock_balances`, movement totals, monthly snapshots, and Stock Hari Ini are derived views over that history.

The core formula is simple:

```text
current_stock = isi + penjualan_reversal - potong - penjualan
```

The meaning behind the terms is where the business rules live. `isi` is manual stock-in. `penjualan` is sales usage written when kitchen marks an item done. `penjualan_reversal` restores sales usage when a completed item is undone. `potong` is manual stock-out or waste, and it stays separate from sales because it has no profit attached to it.

Stock Hari Ini follows current business-cycle semantics. The visible number is current-cycle stock, not accumulated all-time carry-over, while the API can still expose live all-cycle totals for diagnostics and reconciliation. That distinction exists because an owner asking "hari ini" usually wants today's operational stock picture, not a historical balance sheet disguised as a daily view.

## Business Time

The venue does not run on calendar-midnight accounting. Cubic uses a WITA business cycle, defaulting to 07:00 through the next 07:00 as a half-open range. UI copy can show an operator-friendly visible end, but the server filters, cursor reads, diagnostics, and search rebuilds use exact half-open timestamps.

That sounds small until reports are used for real money. Ambiguous date strings, offsetless timestamps, calendar overflows, zero-length ranges, and partial integers are treated as defects rather than conveniences. Owner/admin quick ranges emit exclusive `to` instants so the URL state and API filters agree.

This is also why the owner settings schedule work is careful with date/time semantics. Cycle and shift settings are persisted as operational schedule fields and validated strictly on read; corrupt rows fail loudly rather than silently normalizing into a plausible but wrong schedule.

## Analytics And Histories

Owner and admin analytics are built around bounded reads, URL-owned filters, and shared table components. Date range, search, status, page size, page, and cursor state persist through `nuqs`, while retained TanStack Query data is hidden when it no longer matches the active pagination signature.

The reporting layer is deliberately source-aware:

- Payable money is recalculated from active items, discounts, and stored PPN snapshots.
- Pending totals stay isolated from completed payable totals.
- Best Seller ranks fulfilled item quantity, price, and revenue rather than order-level discount math.
- Best Marka separates no-PPN sales totals from owner-only PPN-included totals.
- Kitchen distribution expands packet components from order-time snapshots.
- Histories use cursor pagination or bounded date windows instead of unbounded all-time reads.

This lets Cubic answer different questions without quietly mixing them together. "What was sold?", "What was wasted?", "Which Marka performed best?", "Which item was cancelled?", and "What stock changed today?" are related questions, but they are not the same query wearing different labels.

## Realtime Operations

Cubic uses process-local SSE for operational freshness. Admin and kitchen live boards receive order invalidations for immediate notification, sounds, latest-order print attempts, and live-head refreshes, while safety polling covers missed events.

Because SSE is process-local, production stays on a single PM2 app instance until there is a distributed realtime bus. The queue is intentionally not treated as pub/sub. It is for durable, idempotent maintenance work such as analytics rollups, order search rebuilds, inventory maintenance, data-growth snapshots, auth cleanup, and monthly stock snapshots.

That distinction keeps the runtime honest:

- SSE is fast but not durable.
- TanStack Query owns client data freshness.
- The maintenance queue owns recoverable background work.
- Operator diagnostics explain when derived models need repair.

## Receipts And Print Agent

Receipt printing became its own subsystem. Admin receipts are internal picking and audit documents, not customer invoices, so they focus on order metadata, item names, quantities, requested drink variants, statuses, and notes rather than price totals.

The browser can auto-print only the latest newly observed order. It stores successful per-order print dedupe locally, retains at most one failed retry candidate, and avoids replaying old receipts on reload, reconnect, or visibility changes. Manual print buttons are separate from auto-print and keep their own duplicate guard.

The local print agent runs on the admin PC at `127.0.0.1:7123`. The browser pairs with it, stores a Cubic-origin token, and sends validated RAW ESC/POS bytes to authenticated `/cubic/receipt-print`. The agent persists jobs and state under `%LOCALAPPDATA%/CubicPrintAgent`, recovers pending spool artifacts, deduplicates same-order same-hash jobs, exposes `/health`, `/version`, `/diagnostics`, and self-test paths, and can run no-printer or Linux emulator modes for development.

This is the kind of unglamorous system work Cubic needed. A receipt printer going offline should not create a hidden backlog of stale receipts that suddenly floods the kitchen later.

## Trust And Access

Cubic is staff-only software with role-aware sessions. Owner, admin, kitchen, and Marka users have different homes, guards, and API access paths. Session cookies are checked against the database, stale cookies are actively expired, and state-changing routes enforce same-origin checks unless they are intentionally public.

Marka quick login is also tied to terminal trust. Owner-managed terminals register browser trust, and production self-hosting can require that trusted terminal before exposing the fast Marka account picker. The owner secret-link flow, `/owner-pin-[password]`, reuses the active owner password rather than introducing a second credential domain.

That trust model is practical rather than theatrical. It is built for a venue where staff phones, browser terminals, and owner recovery paths matter more than enterprise ceremony.

## Deployment And Ops

Cubic is production-oriented. The repo includes bootstrap tooling, a `cubicmanager.sh` operator CLI, standalone Next packaging, PM2 process definitions, Caddy config, MariaDB backup/restore scripts, health checks, staging support, maintenance cron setup, queue worker management, and runbooks under `docs/ops`.

Release flow is artifact-first when needed. A prepared release tarball can be built locally or by GitHub Actions, checksum-verified, transferred to the VPS, started in staging on a dedicated port and database, migrated separately, tested by a human, and only then promoted. Promotion is treated as an operator decision, not an automatic agent habit.

The operational tooling reflects the project’s risk profile:

- Fresh-host bootstrap can deploy from a release artifact.
- Staging has a separate lightweight database and runtime lane.
- Production schema application is explicit.
- Rollback material is preserved.
- Health probes check readiness before declaring success.
- Evlog and request IDs provide searchable incident context.
- Maintenance diagnostics check stock, analytics, partition readiness, queues, and growth.

For a small business app, this is heavy machinery. For a business-critical app that prints orders, records stock, and preserves the owner’s financial history, it is the right kind of boring.

## Engineering Notes

Cubic is built on Next.js App Router, React, TypeScript, Drizzle, MariaDB, TanStack Query, TanStack Table, `nuqs`, Tailwind, Bun, and a set of local scripts that are as much part of the product as the web routes.

The important engineering stance is conservatism. Schema changes go through Drizzle migrations. `db:push` is disabled for normal work. Business logic is pushed into shared helpers where multiple surfaces depend on it. Critical adapters and domain rules have native Bun tests for business cycles, date parsing, order financials, menu sorting, packet stock expansion, stock response semantics, analytics mapping, pagination, auth throttling, and settings validation.

The normal gates are:

```sh
bun run format
bun run fix
bun run typecheck
```

For business-rule changes, `bun test` and `bun run preprod:guardrails` matter too. The latter exercises API auth, business-cycle boundaries, order financials, residual drift diagnostics, and deployment script sanity.

## Reflection

Cubic is the kind of project where the visible UI undersells the system behind it. On the surface, it is staff entering orders and watching tables update. Underneath, it is a ledger, audit trail, realtime feed, local printer bridge, terminal trust model, analytics read-model system, queue worker, and production release machine.

What I like about it is how specific it is. It does not abstract the venue out of existence. It encodes the messy vocabulary of the business: Marka, kosong, potong, Stock Hari Ini, shift cycles, packet components, receipt modes, terminal trust, and owner histories.

That specificity is the product. Cubic works because it is not trying to be universally elegant; it is trying to be operationally correct for one real place.
