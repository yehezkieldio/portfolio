TrashTrack started as a submission for a competition held by the Communication and Informatics Office of the City of Balikpapan. The project reached the finalist stage along with several other submissions, although we did not make it into the final winning top three.

TrashTrack sits in a slightly strange place among my older projects. It is somewhat unique compared to most of the things I usually work on, although parts of it still feel very familiar in hindsight because it remained heavily centered around web technologies and application work.

The project involved Internet of Things (IoT) systems, though I barely touched the IoT side during the project's short lifetime. Most of that work belonged to my friend, who handled the backend systems and the hardware side involving an Espressif ESP board, at least from what I remember.

My side of the project was mostly centered around the TrashTrack mobile application.

## Mobile Development and the Stack

Mobile development was never really my strongest area, and TrashTrack was built under fairly tight time pressure because we were chasing a competition deadline over the course of several weeks.

I originally considered trying React Native with Expo, although that idea disappeared fairly quickly in favor of Capacitor. Looking back, Capacitor was probably the more realistic choice for the situation. It allowed us to stay mostly inside a web development workflow while still shipping an Android application.

The application used React with Vite and Capacitor. Around that, we also used Ionic, shadcn/ui, TanStack Query, React Hook Form, Zod, OpenLayers, Recharts, and i18next.

The Ionic and shadcn/ui combination feels slightly cursed looking back now, although Ionic provided a lot of the mobile-oriented structure we needed quickly enough for the project timeline.

TrashTrack may or may not have been my first real venture into mobile development depending on how strictly that gets defined. It was definitely one of the earlier projects where I had to deal directly with things like camera access, image submission flows, Android packaging, and mobile interaction patterns instead of staying purely inside ordinary browser applications.

The mobile app itself was packaged as a web application synced into Android through Capacitor.

## System Structure

The project had two main audiences.

The first was residents or reporters using the mobile application. They could enter personal details, select a trash bin location, attach a photo, describe a problem, submit a report, and later inspect report history through NIK-based lookup.

The second was operators managing the system itself. The operator side included dashboard charts, trash bin management, sub-bin management, fill history inspection, report processing, feedback creation, and user management.

The system itself was fairly small overall, although the underlying model was more structured than a simple form submission system. Complaints were tied to location, status, reporters, assigned operators, and feedback history. Trash bins were not treated as simple map markers either. They contained sub-containers, capacity-related information, trash events, and historical samples.

A lot of the project's behavior came from trying to preserve those relationships between infrastructure, reports, and operators rather than flattening everything into disconnected records.

## My Work on the Project

My strongest ownership in TrashTrack was the mobile application itself.

Most of my work revolved around screens, navigation, forms, local state management, API consumption, report flows, map displays, operator dashboard surfaces, camera integration, image submission flows, and general application behavior.

OpenLayers handled most of the mapping work, while Recharts was used for dashboard visualizations and charts.

The project also used a monorepo setup where everything lived inside a single repository managed through Nx. That choice mostly came from me. My friend simply went along with it, I think.

My opinion on monorepos today is far more mixed than it was back then. They solve certain coordination problems fairly well, although the ecosystem around them still feels somewhat unresolved even with tooling like Bazel and Turborepo existing now. TrashTrack was still built during the period where I was heavily enthusiastic about Nx-based workspaces and centralized repository structures.

## CI/CD and Automation

One part of the project I still feel fairly proud of is the CI/CD setup.

Using GitHub Actions, I built an automated workflow that compiled the Android application and published signed builds directly into GitHub Releases.

At the time, the automation side honestly felt more intimidating than parts of the application itself. I had to deal with Android signing keys, signing workflows, Java tooling, release builds, and automated artifact publishing while simultaneously learning parts of the Android ecosystem on the fly.

That experience ended up sticking with me more strongly than the mobile development side itself. It was one of the earlier projects where I started genuinely enjoying automation work, release pipelines, and CI/CD systems despite how painful parts of the setup process were.

## Looking Back

TrashTrack is an old project by current standards. The project was built during the Node.js 21 period, which already feels fairly distant compared to how I structure projects now in 2026.

There are many things I would approach differently today, although I still find the project interesting to revisit because it captures a fairly specific stage of how I worked at the time. The stack choices, the monorepo setup, the mobile architecture decisions, and even the way the operator flows were structured all reflect that period fairly clearly.

I am still grateful to my friend for inviting me into the project despite the fact that we lost the competition. I still jokingly complain about it from time to time.

The project also left me slightly traumatized by mobile development, which is probably part of why I still remember it so clearly.
